# Edge Cases — No Donuts

A living log of edge cases to validate together. Each has a **decision** (or `OPEN`) and a **status**.

**Status:** `OPEN` (needs discussion) · `DECIDED` (policy set) · `IMPLEMENTED` (in code + tested)

| # | Edge case | Decision | Status | Owner |
|---|---|---|---|---|
| EC-01 | **On a video call** (another app holds camera) | Try multi-client shared frames; if none, assume PRESENT and don't lock — **bounded** by `maxCallAssumedPresentSeconds` (default 30 min). After that many CONTINUOUS busy-no-frames seconds the assume-present window expires and the engine escalates to absence, so a call app left running unattended eventually runs the normal grace→lock path instead of staying unlocked forever. The cap is NOT the wall-clock time-to-lock: once it expires the engine only *begins* counting absence, so the effective unlock ceiling is `maxCallAssumedPresentSeconds` + the absence consensus (`consecutiveAbsentTicksToLock` ticks) + `graceSeconds`. A genuinely long call with no obtainable frames WILL be locked at the cap — an accepted, bounded fail-open tradeoff (tunable). Any non-busy outcome (real present/absent frame, suspend, unavailable) resets the window — including the production `sessionSuspended()` lock/unlock path, so a fresh call after an unlock starts a clean window rather than re-escalating on a stale one. See [ADR-0003](adr/0003-camera-in-use-policy.md), ND-033. | IMPLEMENTED | blart/homer |
| EC-02 | **Screen already locked / display asleep / session inactive** | Event-driven `SessionStateMonitor` (ND-013, [ADR-0009](adr/0009-session-suspend.md)) pauses the loop and stops the camera while the screen is locked, the display is asleep, or the session is not on the console; resumes both on unlock/wake. On suspend the app calls `PresenceEngine.sessionSuspended()`, which resets absence accounting so resume can't false-lock (the in-tick `.suspended` capture path is a backstop). | IMPLEMENTED | blart/homer |
| EC-03 | **Stranger sits down, enrolled user absent** | **Presence-only MVP (ND-020/025):** detection has no identity, so ANY detected face counts as present — a stranger keeps the Mac unlocked. The strict policy (stranger = ABSENT → lock after grace; a non-matching face must NOT count as present) returns with identity matching in v1.1. | DECIDED (strict behavior deferred to identity) | homer/wiggum/cooper |
| EC-04 | **User present but looking away / turned head** | Detection may miss the face briefly; grace period + consecutive-absent debounce absorbs short gaps. Tune thresholds. **Note (ND-017):** the menu indicator now flips to `.absent` ("away") on the FIRST no-face tick (responsive), while the LOCK still waits for the full consensus + grace window. So a brief glance-away can show "away" transiently in the menu without ever locking — the responsive status and the lock decision are decoupled. | OPEN | cooper/homer |
| EC-05 | **Poor lighting / backlight / glasses / hat / mask** | Affects detection + match. Need robust model + threshold; surface "can't see you" status. | OPEN | cooper |
| EC-06 | **Multiple faces in frame** (user + colleague) | If enrolled user matches any face → PRESENT. Define behavior re: shoulder-surfing later. | OPEN | cooper/homer |
| EC-07 | **External monitor, lid closed (clamshell)** | Built-in camera unavailable → `capture()` returns `.unavailable` → engine surfaces honest `cameraUnavailable` status and does NOT lock (no fail-open; this is now real, not masked by a fake recognizer). User isn't locked out when the only camera is gone. A periodic "can't see you" notification is tracked as ND-045. | DECIDED | blart/homer |
| EC-08 | **Camera permission denied / restricted (MDM)** | `capture()` returns `.unavailable` (denied/restricted or no device); engine surfaces honest `cameraUnavailable` status and does NOT lock. No silent fail-open. krusty owns the prompt/remediation UX; periodic "can't see you" notification tracked as ND-045. | DECIDED | blart/homer/krusty |
| EC-09 | **No camera at all / camera hardware error** | Same policy as EC-07/EC-08: `.unavailable` → `cameraUnavailable`, do NOT lock (never hard-lock the user out when we can't see). Honest visible status; periodic "can't see you" notification tracked as ND-045. | DECIDED | blart/homer |
| EC-10 | **Recognition error / model timeout on a tick** | Conservative HOLD (bounded): a transient `.error` tick neither advances nor resets the absence consensus — no markAbsent (no lock-storm) and no markPresent (no false-unlock); state + counters left untouched until the next clean reading. The 3-tick absence consensus (`consecutiveAbsentTicksToLock`) further damps single-frame glitches. **The hold is bounded:** after `maxConsecutiveErrorsBeforeAbsent` (default 3) CONSECUTIVE errors the engine escalates — it treats the tick as absence (markAbsent), so a wedged recognizer runs the normal grace→lock path and locks rather than holding unlocked forever (no indefinite fail-open). Any clean reading resets the error streak. | DECIDED | homer |
| EC-11 | **User briefly leaves and returns within grace** | No lock; reset cleanly on re-detect. **Note (ND-017):** the menu indicator flips to `.absent` ("away") on the FIRST no-face tick (responsive), but the LOCK waits for the consensus + grace window — so a brief leave-and-return shows "away" transiently in the menu and then clears back to present on re-detect, never locking. New default timing: effective walk-away→lock ≈ consensus (5 × 1s tick) + grace (5s) ≈ 10s. | DECIDED | homer |
| EC-12 | **Photo / phone screen held up to spoof** | Basic anti-spoofing in M4; v1 is not hardened against determined attackers. | OPEN | wiggum/cooper |
| EC-13 | **Laptop sleep / wake, lid open/close** | Handled via `SessionStateMonitor` (ND-013, [ADR-0009](adr/0009-session-suspend.md)): the loop pauses + camera stops on sleep and resumes on wake. On suspend the app calls `PresenceEngine.sessionSuspended()`, which resets absence accounting, so an absence in progress when the Mac sleeps cannot cause a grace-less false lock on wake — the next absence episode rebuilds the full consensus + grace. | DECIDED | blart/homer |
| EC-14 | **Fast user switching / multiple accounts** | Out of scope for v1. Partial mitigation: `SessionStateMonitor` suspends the loop + camera when this session is **not on the console** (`kCGSSessionOnConsoleKey == false`), so we don't run against the login window or another user's session (ND-013). Full multi-account behavior (per-user enrollment, etc.) remains out of scope. | OPEN | homer |
| EC-15 | **App paused by user, then walks away** | Respect pause; don't lock. Consider auto-resume after timed pause. | DECIDED | krusty/homer |
| EC-16 | **Privacy/cover over camera (physical shutter)** | Expected: a covered lens yields a dark frame → Vision returns no faces → `noFace` → counts toward the absence consensus → ABSENT → locks after grace; pending on-device verification. (A distinct "covered" status remains a nice-to-have.) **Note (ND-017):** the menu indicator flips to `.absent` ("away") on the FIRST no-face (covered) tick (responsive), while the LOCK still waits for the consensus + grace window. New default timing: effective cover→lock ≈ consensus (5 × 1s tick) + grace (5s) ≈ 10s. | DECIDED | blart/homer |
| EC-17 | **Presentation / screen sharing without camera** | Camera not in use but user may step away intentionally → normal lock rules apply. Confirm acceptable. | OPEN | homer |
| EC-18 | **Battery / power profile** | Duty-cycle checks; possibly slow tick on battery to save power. | OPEN | blart/homer |
| EC-19 | **Screen-lock request reports success but may not have locked** | The lock **no longer needs Accessibility** ([ADR-0010](adr/0010-screen-lock-no-accessibility.md), supersedes ADR-0006). It is a layered, no-Accessibility mechanism: **(1)** `SACLockScreenImmediate` (resolved at runtime via `dlopen`/`dlsym` from `login.framework`, no link-time dependency), then **(2)** `CGSession -suspend` as a public fallback (login window). `lock()` is **`async`** (`await locker.lock()`) and its verify poll uses `Task.sleep`, so the presence engine's main actor is **never blocked** while a lock is pending. `lock()` **confirms via CGSession** before returning `true`: the screen counts as locked if `CGSSessionScreenIsLocked` is true OR the session is not on the console (`kCGSSessionOnConsoleKey == false`, how `-suspend` presents), read defensively (Bool/NSNumber; on-console signal only trusted when the key is present). If neither mechanism achieves a confirmed lock, `lock()` `os_log`s a warning and returns `false` → engine shows honest `lockFailed`; do **NOT** fail open, and **no stuck fake `.suspended`**. The prior osascript/Ctrl-Cmd-Q path was unreliable (esp. ad-hoc-signed builds) and needed Accessibility — removed. **FUS caveat:** `kCGSSessionOnConsoleKey == false` is ambiguous — it can also mean fast-user-switched-away, not password-locked. It's safe as a lock signal here because `lock()` only runs while ON console (`SessionStateMonitor` pauses the loop off-console, ND-013 / EC-14), so at verify time off-console reliably means our own `CGSession -suspend` succeeded. **Timing:** `lock()`'s verify poll shares ONE ~3s deadline across both mechanisms (SAC then CGSession use the *same* budget, not 3s each), and the poll is cancellation-aware (no CPU busy-spin if the task is cancelled). | DECIDED (verified) | wiggum/homer |

## How to use this file
- Add a row when a new edge case surfaces. Keep it OPEN until we agree on policy.
- When decided, write the policy in the Decision column (link an ADR if architectural).
- When shipped + tested, mark IMPLEMENTED and reference the backlog item.
